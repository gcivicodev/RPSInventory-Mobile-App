Observaciones: El login redirige correctamente a la ruta de sincronización de almacén cuando el usuario tiene flags == 'WW', asegurando que el flujo pase por la pantalla de sync antes de operar (lib/src/views/view_login.dart:38, lib/src/ views/view_sync_almacen.dart:24-89). En la sincronización de almacén primero se suben movimientos e inventario locales y luego se descargan los registros del backend usando la marca de last_sync, con inserciones tipo upsert (lib/src/providers/ sync_provider.dart:173-205, lib/src/providers/sync_provider.dart:585-648, lib/src/views/view_sync_almacen.dart:38-77). La creación de movimientos (tanto normales como “con base en proveedores”) actualiza los saldos locales y guarda el registro en SQLite mediante DBHelper (lib/src/views/view_add_movement.dart:118-181, lib/src/views/ view_add_movement_provider.dart:118-182, lib/src/providers/movement_provider.dart:9-47, lib/src/db_helper.dart:324-502). Los conteos de inventario se almacenan localmente con DBHelper.addInventoryProductCount / updateInventoryProductCount, y se incluyen en la sincronización (lib/src/views/add_inventory.dart:110-189, lib/src/ db_helper.dart:1007-1028, lib/src/db_helper.dart:993-1004, lib/src/db_helper.dart:619). Cambios necesarios: AddInventory envía registros a la API con userId: '1' y username: 'default_user', por lo que el servidor nunca recibe la identidad real del usuario que captura el inventario. Para cumplir el requisito de “tal como se registraron”, hay que tomar userId y username del usuario autenticado (por ejemplo, vía currentUserProvider) antes de crear o actualizar el conteo (lib/src/views/add_inventory.dart:164-165). Tras subir movimientos o inventarios, el código descarga los registros de la nube, pero no hay reconciliación entre el local_id que se envió y el id que devuelve el backend. Esto puede dejar duplicados (un registro con el id    local y otro con el id del servidor) o impedir que el cliente vea el mismo identificador que quedó en la nube (lib/src/db_helper.dart:324-505, lib/src/db_helper.dart:1007-1028, lib/src/providers/sync_provider.dart:585-648, lib/src/models/m_movements.dart:27-99, lib/src/models/m_inventory_products_counts.dart:32-108). Se recomienda, al procesar la respuesta del servidor, localizar el registro original por local_id y actualizarlo (o reemplazar el id local por el del servidor) antes de insertar el remoto, o bien limpiar la tabla y rehidratar sólo con los datos oficiales devueltos por la API.
